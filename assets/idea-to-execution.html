Step 1: Find a Real Problem

Don't start with a solution — start with a pain point.

❌ Solution-First (Bad)

// "I want to build an app that uses
// AI and blockchain to..."

Problem: ???
Users: ???
Why: Because it sounds cool

✅ Problem-First (Good)

// "Students can't find study partners
// for the same course & schedule..."

Problem: Isolation in large classes
Users: University students
Why: Better grades + social connection

🔍 Problem Discovery Techniques

👂 Listen

What do people complain about daily?

🔎 Observe

What workarounds do people use? (spreadsheets, WhatsApp groups, sticky notes)

❓ Ask "Why?" 5 Times

Dig to the root cause, not the symptom

📊 Validate

Talk to 5+ potential users. Do they have this problem?

Step 2: Know Your Users

If you don't know WHO you're building for, you'll build for nobody.

👩‍💻 Sarah 2nd Year CS Student

• Struggles with Data Structures
• Wants study partners for exam prep
• Available evenings & weekends
• Prefers in-person study groups
• Pain: "I don't know anyone in my 200-person lecture"

👨‍🎓 Marcus 3rd Year, Working Part-Time

• Limited schedule (works 20hrs/week)
• Needs flexible, async study options
• Strong in some subjects, weak in others
• Would tutor in exchange for help
• Pain: "My schedule never matches anyone else's"

👩‍🏫 Prof. Chen Course Instructor

• Wants students to collaborate more
• Can't manually matchmake 200 students
• Needs to see engagement metrics
• Would promote tool if it works
• Pain: "I know groups help, but I can't facilitate them"

🎯 User Persona Template

Name: [Give them a real name]
Role: [Who are they?]
Goal: [What are they trying to achieve?]
Pain: [What frustrates them today?]
Context: [When/where/how do they encounter this problem?]
Quote: "[Something they'd actually say]"

Step 3: Write User Stories

Describe features from the user's perspective — not the developer's.

📝 User Story Format

As a [type of user]
I want to [do something]
So that [I get this benefit]

Examples for our Study Group App:

As a student, I want to search for study groups by course, so that I can find people studying the same material.

As a student, I want to see available time slots, so that I can join groups that fit my schedule.

As an instructor, I want to see how many students formed groups, so that I know the tool is helping my class.

✅ Acceptance Criteria

How do you know a story is DONE?

// Story: Search for study groups by course

GIVEN I'm logged in
WHEN I type "CS201" in the search bar
THEN I see all groups for CS201
AND results show group name, members, time
AND I can filter by day of week
AND empty state shows "No groups yet"

🚫 What's NOT a User Story

  • "Use React and PostgreSQL" — that's a tech choice
  • "Make it fast" — too vague, no criteria
  • "Add a database" — that's an implementation task

Step 4: Requirements Document

The blueprint before you write a single line of code.

📄 Requirements Doc Structure

🎯 Problem StatementWhat pain are we solving?
👥 Target UsersPersonas & their needs
✅ Functional ReqsWhat the system DOES
⚡ Non-Functional ReqsPerformance, security, scale
🚫 Out of ScopeWhat we're NOT building (v1)
📊 Success MetricsHow we measure success
📅 TimelineMilestones & deadlines

✅ Functional Requirements (Example)

  • Students can create an account with university email
  • Students can create or join study groups
  • Groups are linked to specific courses
  • Schedule matching shows compatible times
  • In-app messaging for group members

⚡ Non-Functional Requirements

  • Page load < 2 seconds
  • Support 1,000 concurrent users
  • Mobile-responsive design
  • Data encrypted at rest & in transit
  • 99.9% uptime during exam season
07

Step 5: Define Your MVP

The smallest thing you can build to test your idea with real users.

🎯 MoSCoW Prioritization

Must Have Without this, the product doesn't work
Should Have Important but app works without it
Could Have Nice to have, if time allows
Won't Have Explicitly not in this version

💡 MVP Mindset

"If you're not embarrassed by the first version of your product, you've launched too late."

— Reid Hoffman, LinkedIn founder

Study Group App: MVP vs Full Product

Must Have (MVP - Week 1–2)
  • Sign up / log in
  • Browse groups by course
  • Create a group
  • Join a group
Should Have (Week 3–4)
  • Schedule matching
  • Group messaging
  • Notifications
Could Have (Week 5+)
  • AI-powered matching
  • Instructor dashboard
  • Rating system
Won't Have (v1)
  • Video chat integration
  • Payment system for tutors
  • Multi-university support

Step 6: Architecture & Tech Choices

Choose technology based on requirements — not hype.

🌐 Frontend React / Next.js
↔
⚡ API REST / GraphQL
↔
🗄️ Database PostgreSQL
↔
🔐 Auth OAuth / JWT
↔
☁️ Deploy Vercel / Railway

🤔 Decision Framework

Team knows it?Don't pick tech nobody knows for a deadline
Fits scale needs?Don't over-engineer for 100 users
Community/docs?Good docs = faster development
Cost?Students budget ≈ free tier friendly
Hiring pool?Can future team members use this?

🆓 Free-Tier Stack (Student Friendly)

  • Vercel — Frontend hosting + serverless functions
  • Supabase — PostgreSQL + Auth + Realtime
  • GitHub — Version control + CI/CD actions
  • Resend — Transactional emails (free tier)
  • Figma — UI/UX design (free for students)

Step 7: Sprint Planning & Execution

Break it down. Ship in small increments. Get feedback fast.

📅 2-Week Sprint Example

Sprint 1: Foundation

Auth + basic UI + database schema + deploy pipeline

Goal: User can sign up & see empty dashboard
Sprint 2: Core Feature

Create group + browse groups + join group

Goal: User can find & join a study group
Sprint 3: Polish & Launch

Schedule matching + notifications + testing + launch

Goal: MVP live with 10 beta users

📋 Task Breakdown (Sprint 1)

## Sprint 1 Tasks (GitHub Issues)

□ Set up Next.js project + Tailwind
□ Configure Supabase + schema
□ Implement OAuth login (Google)
□ Build navigation component
□ Create dashboard page (empty state)
□ Set up CI/CD with GitHub Actions
□ Deploy to Vercel
□ Write README with setup instructions

Estimated: 15 story points (~2 weeks)

🏃 Daily Standup (2 min each)

1. What did I finish yesterday?

2. What am I doing today?

3. Any blockers?

Step 8: Building — Best Practices

Ship code you won't hate in 3 months.

🌿

Git Workflow

  • Feature branches per task
  • Pull requests for all changes
  • At least 1 code review
  • Meaningful commit messages
  • Never commit to main directly
🧪

Testing

  • Write tests before complex features
  • Unit tests for business logic
  • Integration tests for APIs
  • Manual testing for UI flows
  • Tests run on every PR (CI)
📐

Code Quality

  • Linter + formatter (ESLint, Prettier)
  • TypeScript for type safety
  • Small, focused functions
  • Meaningful variable names
  • No commented-out code in PRs

Step 9: Ship, Measure, Iterate

Launching is not the end — it's the beginning of learning.

🚀 Launch Checklist

  • All MVP features working end-to-end
  • Basic error handling & loading states
  • Mobile responsive
  • Analytics tracking installed
  • Error monitoring (Sentry free tier)
  • Feedback mechanism (even a Google Form)
  • Share with 10 beta users first

📊 Success Metrics

DAU
Daily Active Users
Retention
% back after 7 days
Groups
Created per week
NPS
Would you recommend?

🔄 Feedback Loop

Ship → Observe → Interview → Prioritize → Ship again

Talk to users weekly. Watch what they DO, not just what they SAY.

Common Mistakes to Avoid

Learn from others' failures — it's cheaper.

❌ Mistake

Over-engineering for Day 1

"Let's add microservices, Kubernetes, event sourcing..."

✅ Instead

Start with a monolith

A Next.js app + Supabase handles 90% of use cases to 10K users.

❌ Mistake

Building in stealth for 6 months

"We'll launch when it's perfect..."

✅ Instead

Ship in 2–4 weeks

Get real feedback. Course-correct early. Perfect is the enemy of done.

❌ Mistake

No version control discipline

"I'll just commit everything to main..."

✅ Instead

Branch + PR + Review

Even for solo projects. Future you will thank present you.

❌ Mistake

Skipping user research

"I know what users want, I AM a user..."

✅ Instead

Talk to 5 people minimum

You are not your user. Even 5 conversations reveal blind spots.

Key Takeaways

🎯 Start with WHY

Problem first, solution second. Validate before you build.

👥 Users Are Everything

Talk to them. Build personas. Test with real people early.

📋 Requirements = Clarity

Write it down. If it's not written, it's not agreed upon.

🏃 Ship Small, Ship Fast

2-week MVP > 6-month masterpiece. Learn from real usage.

🔄 Iterate Forever

Launch is Day 1, not the finish line. Keep learning.
