01

From Complaint → Problem Statement

A vague frustration isn't enough. Let's make it specific.

❌ Vague Complaint

"Group projects are annoying"

✅ Clear Problem Statement

[University students] struggle to [find compatible study partners] because [large class sizes make it hard to connect], which leads to [studying alone and lower grades].

🧩 The Formula

[WHO] struggles to [DO WHAT]
because [WHY/BARRIER],
which leads to [NEGATIVE OUTCOME].

🔍 Sharpening Questions

Ask yourself these to go deeper:

  • Who exactly?
    Not "people" — be specific. Students? Which year? Which major?
  • When does this happen?
    Every day? Only during exams? Only in the morning?
  • What do they do today?
    Workarounds = proof the problem is real
  • How bad is it?
    Mild annoyance or "I'd pay to fix this"?
  • Why hasn't someone solved it?
    Maybe they have — go check!

02

Who Is Your User?

You are NOT your user. Even if you have the problem, others experience it differently.

👤 User Persona = A Character Sheet

Give your user a name, a face, a story. It makes every design decision easier.

Name: [A real first name]
Who: [Age, role, context]
Goal: [What they want to achieve]
Frustration: [What blocks them]
Behavior: [How they act today]
Quote: "[Something they'd say]"

🤔 Key Questions to Ask

  • How tech-savvy are they?
  • When & where do they face this problem?
  • What have they already tried?
  • Would they use a phone, laptop, or both?
  • What would make them STOP using your solution?
  • Would they pay for this? Or does it need to be free?

💡 Pro Tip: Find 2 Personas

Your primary user (who uses it most) and a secondary user (who benefits differently). Building for both = wider impact.

03

What Should It Actually DO?

User Stories = describing features from your user's perspective, not yours.

📝 The Format

As a [type of user]
I want to [do something]
So that [I get this benefit]

This forces you to think: WHO wants this, WHAT do they want, and WHY?

🚫 These are NOT user stories:

  • "Use React and Firebase" (tech choice)
  • "Make it responsive" (too vague)
  • "Add a database" (implementation detail)
  • "The system shall..." (robot speak)

As a student, I want to search for study groups by course, so that I find people studying the same thing.

As a busy student, I want to set my available times, so that I only see groups that fit my schedule.

As a group creator, I want to set a max group size, so that the study session stays productive.

As a new user, I want to sign up with my university email, so that I know everyone is a real student.

💡 Tip: Start with the FIRST thing

What does your user do the VERY FIRST time they open your app? That's user story #1.

04

What's Your MVP?

MVP = The smallest version that PROVES your idea works.

🎯 The Rules of MVP

  • Solves ONE core problem well — not 10 problems poorly
  • Can be built in 2–4 weeks — not 6 months
  • Someone can USE it end-to-end — not just see a landing page
  • You can measure if it works — did users achieve their goal?
  • It's OK to be ugly — function > aesthetics for v1

"If you're not embarrassed by v1, you launched too late."

— Reid Hoffman

🔴 Must Have

Without this, the app is useless. These are your MVP.

🟡 Should Have

Important but app works without it. Add in Week 3–4.

🔵 Could Have

Nice to have. Only if you have extra time.

⚪ Won't Have (v1)

Explicitly NOT building this now. Write it down so you don't creep.

💡 The Test

For each story ask: "Can my persona achieve their MAIN goal without this?" If yes → it's NOT a Must Have.

05

Plan the Build

Break it into 1-week sprints. Each sprint ships something usable.

📅 Sprint Thinking

Week 1: Skeleton

Project setup + auth + basic UI + deploy to a URL you can share

✅ End goal: Someone can sign up and see SOMETHING

Week 2: Core Feature

Build the #1 "Must Have" user story end-to-end

✅ End goal: User can do THE THING your app is for

Week 3: Polish + Launch

2nd Must Have + error handling + mobile + share with 10 people

✅ End goal: Real people using it and giving feedback

🧱 Breaking Stories → Tasks

Each user story becomes 3–6 tasks:

## Story: "User can search groups by course"

□ Design search UI (input + results list)
□ Create database table for groups
□ Build API endpoint: GET /groups?course=X
□ Connect frontend to API
□ Handle empty state ("No groups yet")
□ Test with sample data

⏰ Time Estimation Trick

Think how long it'll take. Now multiply by 2. That's closer to reality. First-time estimates are always optimistic.

06

How to Know If It's Working

Ship it. Watch what happens. Talk to users. Repeat.

📊 Define Success BEFORE You Build

If you don't decide what "working" means upfront, you'll never know if you succeeded.

// Pick 2–3 metrics that matter:

Metric 1: Do users sign up?
Target: 20 signups in first week

Metric 2: Do they DO the thing?
Target: 50% create/join a group

Metric 3: Do they come back?
Target: 30% return within 7 days

🔄 The Feedback Loop

🚀 Ship
→
👀 Observe
→
🗣️ Ask
→
🔧 Improve

🗣️ 5 Questions for Beta Users

  1. What did you expect to happen?
  2. What confused you?
  3. Would you use this again? Why/why not?
  4. What's the ONE thing you'd change?
  5. Would you recommend this to a friend?
